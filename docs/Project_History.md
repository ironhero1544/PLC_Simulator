# 프로그램 제작 변천사

## 프로그램 제작의 발단

### PLC를 만나다.

저는 2025년 4월 반도체 설비 보전 기능사 자격증을 준비하며 PLC를 처음 보게 되었습니다.
저는 항상 PC와 관련된 코딩만 해본지라 PLC 코드를 처음 접하게 되었고 LD, FBC, ST 같은 레더 언어를 익힐 수 있었습니다.

![반설보 작업.jpg](../docs_img/%EB%B0%98%EC%84%A4%EB%B3%B4%20%EC%9E%91%EC%97%85.jpg)

하지만 PLC 작업대 수에 비해 자격증을 준비하는 인원의 수가 너무 많아 교대로 연습을 하게 되었습니다.
교대로 하게 되면 시간이 너무 많이 줄어들어 차라리 내가 사서 준비하지 하며 찾아보니
아뿔사 개인이 사기에는 너무나 비싸고 구매하기도 어려운 것이었습니다.
결국 이를 받아들여 연습 차례가 되면 쉬지도 않고 열심히 연습하였습니다.
하지만 충분한 연습시간 없이 시험을 보게 되어 매우 어려운 시험이 되었습니다.
PLC 작업이 어려운 것이 아닌 충분한 연습을 하지 않아 일어나는 잔 실수로 인해 시간을 아슬아슬하게 맞추게 된 것이었습니다.
PLC 실습은 장비 의존도가 높아 개인이 충분히 반복 학습하기 어려운 구조적인 한계가 있었습니다.
저는 이런 경험으로 PLC 작업을 충분하게 연습할 수 있는 솔루션을 고민하게 되었습니다.

### 프로그램 만들기 수행평가

![프로그램 제작 발단.png](../docs_img/%ED%94%84%EB%A1%9C%EA%B7%B8%EB%9E%A8%20%EC%A0%9C%EC%9E%91%20%EB%B0%9C%EB%8B%A8.png)

저의 학교에서는 2학년에 프로그래밍 과목이 있어 C언어의 기초를 배울 수 있었습니다.
그중에서 마지막 수행평가는 C언어를 이용한 프로그램 제작이라는 응용 수행평가였습니다.
저는 이전에도 많은 SW 대회 등에 참여하였고 프로그래밍을 이전에도 해본 경험이 있었습니다.
저의 C언어 능력을 사용해서 수행평가에서 좋은 평가를 받을 수 있는 아이디어를 고민하다,
이전 PLC 작업에서 고민했던 솔루션을 생각하여,
**실제 PLC 작업을 컴퓨터를 활용하여 그대로 재현할 수 있으면, PLC가 없어도 컴퓨터로 PLC 작업을 연습할 수 있지 않을까?**
생각하게 되었습니다.
그래서
PLC 프로그래밍 환경(GX Works2), 전자 부품 회로 시뮬레이션(TinkerCAD)의 요소에서 착안하여
하나의 프로그램에서 **실제 배선과 프로그래밍을 하고 작동까지 할 수 있는 시뮬레이션 프로그램**을 만들기 시작했습니다.

## MVP(Minimum Viable Product) 작업 시작

![개발 시작.png](../docs_img/%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91.png)

먼저 저는 C언어 개발을 해보기는 했지만,
아두이노나 마이크로 컨트롤러 등의 하드웨어 제어에 들어가는 프로그램만 만들어 와서
GUI 기반 윈도우 프로그램을 만들어 보지 않아 UI 제작에 큰 어려움을 느꼈습니다.

그래서 먼저 AI를 사용해서 제가 추가하고 싶은 요소와 UI 및 기본 작동을 Claude의 아티팩트 기능을 사용해서 먼저 구현했습니다.
[실배선 모드.tsx](%EC%8B%A4%EB%B0%B0%EC%84%A0%20%EB%AA%A8%EB%93%9C.tsx)
[프로그래밍 모드.tsx](%ED%94%84%EB%A1%9C%EA%B7%B8%EB%9E%98%EB%B0%8D%20%EB%AA%A8%EB%93%9C.tsx)

이렇게 하여 완성된 UI를 기반으로 실제 C언어 기반 프로그램을 제작하였습니다.
이 UI를 완성한 날은 수행평가 제출 7일 전이었습니다.
학교 일정들과 겹치고 **AI를 잘 다루는 것에 서툴렀기에** 많은 시간이 소요되었습니다.

그리고 명확하게 코드 형태로 제작된 리액트 UI 코드였지만
실제 C언어로 만드는 과정은 그리 쉽지 않았습니다.

C언어는 리액트에서 사용하는 타입스크립트 언어와 차이가 심해도 많이 심했습니다.
C언어의 추상화 수준과 타입스크립트의 추상화 수준의 차이가 심하고
IMGUI로 인해 강제로 C++로 컴파일하여 사용하였는데
**IMGUI가 extern "C"에서 사용할 수 없어 C++로 코드를 다시 바꾸게 되는 등**
MVP를 완성하는데 많은 시행착오를 겪게 되었습니다.

결국 수행평가 제출 5분 전 불완전한 상태로 제출하였지만
**프로그램이 너무 무거워 업로드에 10분이 지체되어
제시간에 제출하지 못해 수행평가를 날려먹었습니다...**
![프로그램 제출.png](../docs_img/%ED%94%84%EB%A1%9C%EA%B7%B8%EB%9E%A8%20%EC%A0%9C%EC%B6%9C.png)

다행히 선생님이 선처해 주셔 미제출까지 점수가 깎이지 않았지만 프로그램이 완성에 미치지 않았고
목표했던 수행평가 만점도 아쉽게도 실패하면서 좌절하였습니다.

제가 이 프로그램 개발을 열심히 진행했다는 걸 아는 다른 이들과 선생님은 저를 격려하였지만
저는 프로그램을 완성시키지 못한 것에 큰 아쉬움을 느꼈습니다.

아쉬움을 느낀 뒤 미완성된 MVP 프로그램이라도 수행평가는 발표까지 진행해야 하는 것이었기에
제가 만든 프로그램을 다른 이에게 발표하게 되었습니다.
![MVP개발 후 발표 경험.jpg](../docs_img/MVP%EA%B0%9C%EB%B0%9C%20%ED%9B%84%20%EB%B0%9C%ED%91%9C%20%EA%B2%BD%ED%97%98.jpg)
![MVP 프로그램 UI.png](../docs_img/MVP%20%ED%94%84%EB%A1%9C%EA%B7%B8%EB%9E%A8%20UI.png)

발표를 하면서 제가 기대했던 것보다 반응이 썩 좋지는 않았습니다.

MVP 프로그램의 완성도가 떨어지기도 하고 다들 프로그램이 **정확히 무엇을 하는지도 모르는 분위기**였습니다.

참담하게도 MVP 프로그램의 결과는 좋지 않았고 **이 프로그램은 시험과 제가 준비하던 다른 일들로 인해 미뤄져 있었습니다...**

## 프로그램을 싹 갈아 엎다

### 문제점 분석

쓴 실패를 뒤로하고 시험도 끝나고 준비하던 것도 일단락된 학기말
**저는 이전 MVP에서 어떤 문제가 있었는지 나열했습니다.**

1. 코드 작성에서 정확한 스타일 없이 진행하여 수정하기 힘들게 되어 있다.
2. 한 파일에 너무 많은 코드가 작성되어 있다.
3. 실배선과 프로그래밍 모드가 서로 분리되어서는 잘 작동하지만 연결을 할 수 없었다.
4. PLC 작동에 대한 이해도가 부족하여 실제와 다른 작동을 하는 일이 많았다.
5. AI를 너무 멍청하게 사용했다.

더 많은 문제가 있었지만 크게 보자면 이렇게 5개였습니다.
이러한 문제를 해결하기 위해 **전체 코드를 리팩토링 하는 것을 결정하였습니다.**

### 코드 스타일 결정

MVP 때도 UI를 미리 준비하긴 하였지만
프로젝트 초반에 해야 했던 작업이 덜 되어 있어 MVP 코드는 더러운 코드 뭉치가 되었습니다.

저는 이러한 일이 반복되지 않도록 **코드 스타일을 먼저 [GOOGLE CPP STYLE](https://google.github.io/styleguide/cppguide.html)에 맞춰 작성하였습니다.**
여러 가지 코드 스타일이 있었지만 다양한 프로젝트가 진행되는 구글에서 사용하는 방법이 가장 범용성 있고
이 프로젝트에 맞다고 생각하여 프로젝트의 코드 스타일로 결정하였습니다.

### 디렉토리 구조화

MVP에선 전 하나의 파일에 너무 많은 코드를 작성하여 코드 작성에 큰 어려움을 느꼈습니다.
그래서 **디렉토리 구조를 더 세분화하여 코드 수정을 보다 더 편리하게** 하였습니다.

### 새로운 물리 엔진 구상 및 OPENPLC 엔진 적용

실배선과 프로그래밍 모드의 연결을 위해서
PLC 연결 방식을 물리 엔진과 함께 고민하였습니다.
**먼저 MVP에서 만든 조잡한 물리 엔진을 과감하게 버리고 처음부터 다시 물리 엔진을 구상하였습니다.**
물리 현상을 분리하여 **전기, 공압, 기계로 나누었고 이 3가지 현상을 별도로 작동하게 하여**
**복잡하게 작동하지 않으면서 현실성 있게 변경**하였습니다.

또한 프로그래밍 모드에서 PLC 레더를 해석하고 이걸 실배선 환경에 적용시키는 방식을 새롭게 바꾸었습니다.
기존에는 레더 해석한 걸 그대로 실배선에 적용하는 방식으로 시도했습니다.
레더 해석도 문제였지만 실배선에 적용하는 과정에서 각종 버그와 성능 문제가 발생했습니다.
PLC 공부를 아예 안 하지는 않았지만 제가 레더 해석 및 실행기를 만들 정도가 되지도 않았고 만들기에는 너무 오랜 시간이 필요하기에
**PLC 관련 오픈소스 프로젝트 중 하나인 [OPENPLC 편집기](https://github.com/thiagoralves/OpenPLC_Editor)의 레더 해석기와 컴파일러를 기반으로** 프로그램에 맞게 튜닝하여
**프로그래밍 모드 레더 작성 -> 레더 해석 -> C++ 코드로 변환 -> 실배선 환경에서 에뮬레이팅하여 C++ 프로그램 작동 -> PLC 작동**
이러한 구조로 **실제 PLC 작동과 같은 환경을 구성**하였습니다.

### MD 문법과 구체화된 계획으로 효율적인 AI 사용

저는 이전에는 AI를 사용하면서 구체적인 계획을 작성하지 않았고 항상
xx을 구현해줘, xx이 안돼, 컴파일 오류남
이런 식으로만 사용하였습니다.

이 방식으로는 **제가 원하는 대로 작동하지 않는다는 것을 MVP를 제작하면서 뼈저리게 느꼈습니다.**
그래서 **MD 문법과 구체적인 계획을 작성하여 여러 번 검토를 하여 효율적으로 AI를 사용**하였습니다.

이전 MVP에서 발생했던 **구조적 문제를 해결하고 실제로 동작 가능한 PLC 시뮬레이션 프로그램**으로 발전시킬 수 있었습니다.

## 선생님들의 피드백 및 새로운 기능 추가와 완성도 높이기

리팩토링이 끝나고 새 학기가 시작되어 학기말까지 이 프로젝트 말고도 많은 일이 있어 긴 시간이 지났습니다.
학기말 다시 프로젝트를 시작하며 선생님들에게 피드백을 받기 시작했습니다.

PLC 전문가 선생님들과 기타 선생님들에게 공통으로 받은 피드백으로는
반도체 설비 보전 기능사뿐만 아닌 자동화 설비 기능사 부품도 추가하라는 것이었습니다.

### 성능 최적화

기능이 추가되고 부품이 많아질수록, 특히 **자동화 설비 기능사**의 복잡한 기구물들이 추가되면서 프로그램이 점점 무거워지는 문제가 발생했습니다.
여러 개의 센서와 실린더, 그리고 많은 배선이 동시에 작동할 때 렉이 걸리면 시뮬레이션으로서의 가치가 떨어진다고 생각했습니다.

기존에 구축한 **물리 엔진의 구조는 유지하면서**, 더 부드럽게 돌아가도록 연산 결과는 유지하면서 중간 연산을 줄이는 식으로 최적화를 진행했습니다.

1. LOD 시스템
   캔버스 카메라의 **Zoom 값에 따라 물리 연산의 정밀도를 다르게** 했습니다.
   가까이 볼 때는 아주 미세한 움직임까지 계산하여 부드러운 애니메이션을 보여줍니다.
   멀리 볼 때는 전체적인 흐름만 파악하면 되므로, 물리 연산 주기를 조절하여 CPU 부하를 줄였습니다.

2. View Frustum Culling - 뷰 절두체 분리
   부품이 아무리 많아도 사용자가 보고 있는 화면은 한정적입니다. 그래서 렌더링 단계에서 **화면 밖에 있는 부품과 배선은 그리기 연산을 아예 생략**하도록 만들었습니다.
   덕분에 부품이 수백 개가 되어도 그래픽 처리를 줄여 안정적으로 프로그램이 동작하도록 하였습니다.

3. 공간 분할 및 Box2D 최적화
   공작물이 컨베이어나 실린더와 부딪히는 걸 계산할 때, 모든 부품을 하나하나 다 검사하니 속도가 너무 느려졌습니다.
   이를 해결하기 위해 **공간을 Grid로 나누어, 내 주변에 있는 부품하고만 충돌 검사를 수행**하도록 바꿨습니다. 또한, 움직이지 않는 공작물은 연산에서 제외시켜 불필요한 연산을 없앴습니다.
   또한 공작물 충돌 및 센서 인식은 오픈 소스인 BOX2D 물리엔진으로 처리하여 더 효율적인 충돌 연산으로 변경하여
   연속적인 충돌에서의 스로틀링이 일어나는 현상을 해결할 수 있었습니다.

4. 멀티스레딩
   **서로 상관없는 부품들의 상태 업데이트(전기 신호, 밸브 로직 등)를 여러 스레드가 나눠서 병렬로 처리**하도록 멀티스레딩을 구현했습니다.
   부품이 많아질수록 처리 속도에서 큰 이득을 볼 수 있었습니다.

5. 메모리 풀링
   물리 엔진 내부에서 자주 생기고 사라지는 전기 노드나 공압 엣지 같은 객체들은
   **미리 메모리를 할당해 두고 재사용하는 메모리 풀 기법**을 적용하여 런타임 오버헤드를 최소화했습니다.

이러한 최적화로 기존에는 물리 작업 중에 6fps까지 떨어지던 것이
i5 3470 DDR3 8GB GT1030의 저사양 환경에서 프레임 제한을 해제하니 400fps대까지... 나오게 되었습니다.
