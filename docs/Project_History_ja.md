# プログラム制作変遷史

## プログラム制作のきっかけ

### PLCとの出会い

私は2025年4月、半導体設備保全技能士資格の準備をしながら、初めてPLCに触れることになりました。
私はこれまでPCに関連するコーディングしかしたことがなかったため、PLCコードを初めて接することになり、LD、FBC、STのようなラダー言語を学ぶことができました。
![半導体設備保全作業.jpg](../docs_img/%EB%B0%98%EC%84%A4%EB%B3%B4%20%EC%9E%91%EC%97%85.jpg)

しかし、PLC作業台の数に比べて資格を準備する人数が多すぎて、交代で練習をすることになりました。
交代でやると時間が大幅に減ってしまうので、いっそ自分で購入して準備しようと思い調べてみると、
なんと個人が購入するには高すぎて、購入するのも難しいものでした。
結局これを受け入れて、練習の順番が来たら休みもせず一生懸命練習しました。
しかし、十分な練習時間なしに試験を受けることになり、非常に難しい試験となりました。
PLC作業が難しいのではなく、十分な練習をしていなかったために起こる小さなミスで時間をギリギリ合わせることになったのです。
PLC実習は機器への依存度が高く、個人が十分に反復学習するのが難しい構造的な限界がありました。
私はこのような経験から、PLC作業を十分に練習できるソリューションを考えるようになりました。

### プログラム制作のパフォーマンス評価

![プログラム制作のきっかけ.png](../docs_img/%ED%94%84%EB%A1%9C%EA%B7%B8%EB%9E%A8%20%EC%A0%9C%EC%9E%91%20%EB%B0%9C%EB%8B%A8.png)

私の学校では2年生にプログラミング科目があり、C言語の基礎を学ぶことができました。
その中で最後のパフォーマンス評価は、C言語を利用したプログラム制作という応用パフォーマンス評価でした。
私は以前にも多くのSWコンテストなどに参加しており、プログラミングをした経験がありました。
私のC言語能力を使ってパフォーマンス評価で良い評価を受けられるアイデアを考えていたところ、
以前のPLC作業で考えていたソリューションを思い出し、
**実際のPLC作業をコンピュータを活用してそのまま再現できれば、PLCがなくてもコンピュータでPLC作業を練習できるのではないか？**
と考えるようになりました。
そこで、
PLCプログラミング環境（GX Works2）、電子部品回路シミュレーション（TinkerCAD）の要素からヒントを得て、
一つのプログラムで**実際の配線とプログラミングをし、動作までできるシミュレーションプログラム**を作り始めました。

## MVP（Minimum Viable Product）作業開始

![開発開始.png](../docs_img/%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91.png)

まず、私はC言語の開発をしたことはありましたが、
Arduinoやマイクロコントローラーなどのハードウェア制御に入るプログラムだけを作ってきたので、
GUIベースのWindowsプログラムを作ったことがなく、UI制作に大きな困難を感じました。

そこで、まずAIを使って私が追加したい要素とUI及び基本動作をClaudeのアーティファクト機能を使って先に実装しました。
[実配線モード.tsx](%EC%8B%A4%EB%B0%B0%EC%84%A0%20%EB%AA%A8%EB%93%9C.tsx)
[プログラミングモード.tsx](%ED%94%84%EB%A1%9C%EA%B7%B8%EB%9E%98%EB%B0%8D%20%EB%AA%A8%EB%93%9C.tsx)

このようにして完成したUIを基に、実際のC言語ベースのプログラムを制作しました。
このUIを完成させた日は、パフォーマンス評価提出の7日前でした。
学校の日程と重なり、**AIをうまく扱うことに不慣れだった**ため、多くの時間がかかりました。

そして、明確にコード形態で制作されたReact UIコードでしたが、
実際にC言語で作る過程はそれほど簡単ではありませんでした。

C言語はReactで使用するTypeScript言語との違いが非常に大きかったです。
C言語の抽象化レベルとTypeScriptの抽象化レベルの差が激しく、
IMGUIのために強制的にC++でコンパイルして使用したのですが、
**IMGUIがextern "C"で使用できず、C++でコードを書き直すことになるなど**、
MVPを完成させるまで多くの試行錯誤を経験しました。

結局、パフォーマンス評価提出5分前に不完全な状態で提出しましたが、
**プログラムが重すぎてアップロードに10分遅れてしまい、
期限内に提出できずにパフォーマンス評価を台無しにしてしまいました...**

私がこのプログラム開発を一生懸命進めていたことを知っている他の人や先生は私を励ましてくれましたが、
私はプログラムを完成させられなかったことに大きな悔しさを感じました。

悔しさを感じた後、未完成のMVPプログラムでもパフォーマンス評価は発表まで進めなければならなかったので、
私が作ったプログラムを他の人に発表することになりました。

## プログラムを全面的に作り直す

### 問題点の分析

苦い失敗を経て、試験も終わり、準備していたことも一段落した学期末、
**私は以前のMVPでどんな問題があったのかを列挙しました。**

1. コード作成において正確なスタイルなしに進めたため、修正が難しくなっていた。
2. 一つのファイルにあまりにも多くのコードが書かれていた。
3. 実配線とプログラミングモードがそれぞれ分離されてはよく動作したが、接続することができなかった。
4. PLC動作に対する理解度が不足していて、実際とは異なる動作をすることが多かった。
5. AIをあまりにも愚かに使用していた。

もっと多くの問題がありましたが、大きく見るとこのように5つでした。
このような問題を解決するために、**全体のコードをリファクタリングすることを決定しました。**

### コードスタイルの決定

MVPの時もUIを事前に準備しましたが、
プロジェクト初期に行うべきだった作業が不足していて、MVPコードは汚いコードの塊になっていました。

私はこのようなことが繰り返されないように、**コードスタイルをまず[GOOGLE CPP STYLE](https://google.github.io/styleguide/cppguide.html)に合わせて作成しました。**
様々なコードスタイルがありましたが、様々なプロジェクトが進行されるGoogleで使用される方法が最も汎用性があり、
このプロジェクトに合っていると思い、プロジェクトのコードスタイルとして決定しました。

### ディレクトリ構造化

MVPでは一つのファイルにあまりにも多くのコードを書いて、コード作成に大きな困難を感じました。
そこで、**ディレクトリ構造をさらに細分化してコード修正をより便利に**しました。

### 新しい物理エンジンの構想及びOPENPLCエンジン適用

実配線とプログラミングモードの接続のために、
PLC接続方式を物理エンジンと共に検討しました。
**まず、MVPで作った粗末な物理エンジンを思い切って捨てて、最初から物理エンジンを再構想しました。**
物理現象を分離して**電気、空圧、機械に分け、この3つの現象を別々に動作させることで**、
**複雑に動作せず現実性を持つように変更**しました。

また、プログラミングモードでPLCラダーを解釈し、それを実配線環境に適用する方式を新しく変えました。
既存はラダーを解釈したものをそのまま実配線に適用する方式で試みました。
ラダー解釈も問題でしたが、実配線に適用する過程で様々なバグと性能問題が発生しました。
PLC学習を全くしなかったわけではありませんが、私がラダー解釈及び実行機を作るほどのレベルではなく、作るにはあまりにも長い時間が必要だったので、
**PLC関連のオープンソースプロジェクトの一つである[OPENPLCエディタ](https://github.com/thiagoralves/OpenPLC_Editor)のラダー解釈機とコンパイラをベースに**、プログラムに合わせてチューニングし、
**プログラミングモードでラダー作成 -> ラダー解釈 -> C++コードに変換 -> 実配線環境でエミュレートしてC++プログラム動作 -> PLC動作**
このような構造で**実際のPLC動作と同じ環境を構成**しました。

### MD文法と具体化された計画による効率的なAI使用

私は以前はAIを使用しながら具体的な計画を作成せず、常に
xxを実装して、xxができない、コンパイルエラーが出る
このような方式でのみ使用していました。

この方式では**私が望むように動作しないということをMVPを制作しながら痛感しました。**
そこで、**MD文法と具体的な計画を作成して何度も検討し、効率的にAIを使用**しました。

[REFACTORING_PLAN.md](../legacy/docs/.copilot/REFACTORING_PLAN.md)
[DIRECTORY_STRUCTURE.md](../legacy/docs/.copilot/DIRECTORY_STRUCTURE.md)

このような構造改善を通じて、**わずか2週間でプロジェクトを全面的にリファクタリング**し、
以前のMVPで発生した**構造的問題を解決し、実際に動作可能なPLCシミュレーションプログラム**に発展させることができました。

### パフォーマンス最適化

機能が追加され、部品が増えるにつれて、特に**自動化設備技能士**の複雑な機構が追加されると、プログラムがどんどん重くなる問題が発生しました。
複数のセンサーとシリンダー、そして多くの配線が同時に動作するとラグが発生し、シミュレーションとしての価値が落ちると思いました。

既存に構築した**物理エンジンの構造は維持しながら**、よりスムーズに動くように演算結果は維持しながら中間演算を減らす方式で最適化を進めました。

1. LODシステム
   キャンバスカメラの**Zoom値に応じて物理演算の精度を変える**ようにしました。
   近くで見る時は非常に微細な動きまで計算してスムーズなアニメーションを見せます。
   遠くで見る時は全体的な流れだけを把握すればいいので、物理演算周期を調整してCPU負荷を減らしました。

2. View Frustum Culling - ビューフラスタム分離
   部品がどれだけ多くても、ユーザーが見ている画面は限られています。そこで、レンダリング段階で**画面外にある部品と配線は描画演算を完全にスキップ**するようにしました。
   おかげで部品が数百個になっても、グラフィック処理を減らして安定的にプログラムが動作するようにしました。

3. 空間分割及びBox2D最適化
   工作物がコンベアやシリンダーと衝突するのを計算する時、すべての部品を一つ一つチェックすると速度が遅すぎました。
   これを解決するために、**空間をGridに分けて、自分の周りにある部品とだけ衝突検査を実行**するように変えました。また、動かない工作物は演算から除外して不要な演算をなくしました。
   また、工作物の衝突とセンサー認識はオープンソースのBox2D物理エンジンで処理し、より効率的な衝突演算に変更して、
   連続的な衝突でのスロットリングが起こる現象を解決できました。

4. マルチスレッディング
   **互いに関係のない部品の状態更新（電気信号、バルブロジックなど）を複数のスレッドが分けて並列に処理**するようにマルチスレッディングを実装しました。
   部品が多くなるほど処理速度で大きな利益を得られました。

5. メモリプーリング
   物理エンジン内部で頻繁に生成・消滅する電気ノードや空圧エッジのようなオブジェクトは、
   **あらかじめメモリを割り当てておいて再利用するメモリプール技法**を適用して、ランタイムオーバーヘッドを最小化しました。

このような最適化により、既存は物理作業中に6fpsまで落ちていたものが、
i5 3470 DDR3 8GB GT1030の低スペック環境でフレーム制限を解除すると400fps台まで...出るようになりました。

## 終わりに

久しぶりの一人でのプロジェクトです。
いつもチームメンバーとのプロジェクトはコンテストに参加するものだったので、
私が考える完成とはかけ離れた非常に粗末な作品でしたが、締め切りを考えずに作ると、
ようやく私が考える完成に近い作品ができて、個人的にはとても嬉しいです。
チームメンバーとのプロジェクトはコーディング中心というよりアイデアを実現するものでしたが、
このプロジェクトはコーディング中心で、私が実際に感じた不便さを解消しながらも、
新しい技術や理論などを経験として学びながら、最後まで完成させるための自分との戦いだったように思います。
私が最も得意とする分野で、思う存分作れるプロジェクトでした。
他人の助けが全くなかったわけではありませんが、コードは私一人で書き、デザインまで一人で作業しました。
プロジェクトはオープンソースとして公開し、部品追加や他のアップデートは時間がある時か、素晴らしいプルリクエストが来ない限り、
これ以上進める時間がないと思います。その後は、このプログラムの目的から大きく外れることになりそうです。
以上です。終わり！
